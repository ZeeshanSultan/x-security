# x-security schema — current field reference (v0.8)

Schema id: `https://usewaf.com/schemas/x-security/v0.8.json`
Source of truth: `packages/schema/src/x-security.schema.json` (enums, `$defs`, regexes),
`packages/schema/src/types.ts` (TS types), `packages/schema/src/owasp-mapping.json`
(category → mitigating-field attributions).

Status: canonical current field reference. **Supersedes `v0.3-additions.md`**, which
documents only the v0.3 delta and predates every field added in v0.4–v0.8. The LLM
detection prompt advertises this schema id (`SCHEMA_REF` in
`packages/llm-agent/src/agentic/prompts.ts`); it must name the *current* version so the
model knows the v0.4–v0.8 fields exist and is allowed to emit them. Pointing the model
at v0.3 was a P0 detection bug — the model never emitted any field newer than v0.3.

This doc is a reference for the fields **added in v0.4 through v0.8**. The v0.1–v0.3
core (authentication, authorization, csrf, rateLimit, request/response schema, headers,
graphql, websocket, botProtection, etc.) is unchanged; see `v0.3-additions.md` for the
v0.3 delta and `x-security.schema.json` for the full grammar. Everything below is
additive — existing policies continue to validate.

> **Enforcement statuses** quoted here come verbatim from the schema-JSON field
> descriptions. Where a field's enforcement varies by target (kong / coraza / envoy /
> bunkerweb / openappsec / cloudflare / aws-apigw / firewall), the per-target detail
> lives in the compiler capability matrix — this doc states the schema-declared posture
> and defers exact per-target cells to "see capability matrix".

## 1. Security category ids — OWASP-API + x-security-native (SSEC)

Two parallel id spaces, kept structurally separate so the 10 pure OWASP-API-2023 cells
can never be overloaded (Rule D-1: overloading e.g. `API8` would silently corrupt every
existing `API8` finding).

- **`OwaspId`** — pattern `^API([1-9]|10):2023$`. Hand-authored on `mitigates` arrays.
- **`SsecId`** — pattern `^SSEC-(INJECTION|PROMPT|AUDIT|STORAGE)$`. **Never hand-authored
  on `mitigates`**; SSEC attribution is *derived* by probing capabilities (e.g.
  `request.schema.<field>.injectionGuard`), so the OWASP regex stays pure.
- **`SecurityCategoryId`** = `OwaspId | SsecId` — the union reporters key on.

### The 4 SSEC-native attributions (from `owasp-mapping.json`)

| SSEC id | name | x-security field that drives it (`mitigatedBy`) |
|---|---|---|
| `SSEC-INJECTION` | Injection (x-security-native) | `request.schema.injectionGuard` (all sinks except `ai-prompt`) |
| `SSEC-PROMPT` | LLM Prompt Injection (x-security-native) | `request.schema.injectionGuard` (sink `ai-prompt`) |
| `SSEC-AUDIT` | Audit Logging (x-security-native) | `logging` |
| `SSEC-STORAGE` | At-Rest Storage Posture (x-security-native, advisory-only) | `request.dataAtRest` |

These classes exist because OWASP-API-2023 does not enumerate Injection, LLM
prompt-injection, audit logging, or at-rest storage posture as dedicated API cells.

## 2. `request.schema.<field>.injectionGuard` — per-arg injection hardening

**Field path:** `request.schema.<field>.injectionGuard` (on `ParamSchema`).
**Introduced:** v0.6 (W19); **extended v0.7** with `deserialization` + `ai-prompt`.

**Type:** array (unique) of sinks —
`('sql' | 'nosql' | 'os-command' | 'xpath' | 'ldap' | 'code-eval' | 'xss' | 'deserialization' | 'ai-prompt')[]`
(9 sinks).

Opt-in, per-argument, explicit. Declares which injection sink(s) the field flows into so
the compiler emits a native operator on `ARGS:<field>|ARGS:json.<field>`. This replaces
the implicit "string-type heuristic" — it is per-arg and explicit.

**Sink → operator (from schema description), and SSEC attribution:**

| sink | gateway operator (per schema doc) | SSEC category |
|---|---|---|
| `sql` | `@detectSQLi` | `SSEC-INJECTION` |
| `nosql` | operator-token denylist | `SSEC-INJECTION` |
| `os-command` | shell-metachar denylist | `SSEC-INJECTION` |
| `xpath` | injection-token denylist | `SSEC-INJECTION` |
| `ldap` | injection-token denylist | `SSEC-INJECTION` |
| `code-eval` | code-eval token denylist | `SSEC-INJECTION` |
| `xss` | `@detectXSS` | `SSEC-INJECTION` |
| `deserialization` | unsafe-deserialization preamble denylist (node-serialize / Java `rO0` / PHP `O:<n>:` / python pickle) | `SSEC-INJECTION` |
| `ai-prompt` | LLM prompt-injection heuristic denylist | **`SSEC-PROMPT`** |

**Attribution rule:** all sinks **except** `ai-prompt` → `SSEC-INJECTION`; `ai-prompt` →
the distinct `SSEC-PROMPT` class. **Never an OWASP-API cell.**

**Enforcement:** edge-enforceable on **coraza / bunkerweb only** (per schema description).
Per-target detail: see capability matrix.

## 3. `logging` — top-level audit/access logging policy

**Field path:** `logging` (top-level on `XSecurityPolicy`).
**Introduced:** v0.7.
**SSEC category:** `SSEC-AUDIT` (never an OWASP-API cell — OWASP-API-2023 retired the
dedicated logging class).

**Type — `Logging`:**
```ts
interface Logging {
  events: LoggingEvent[];          // required
  sink?: LoggingSink;              // defaults to 'stdout'
  sinkRef?: VarRef;                // required when sink === 'http-collector'
  piiRedaction?: boolean;          // mask/omit declared pii fields; default false
}
type LoggingEvent =
  | 'auth-failure' | 'auth-success' | 'authz-deny'
  | 'rate-limit-trip' | 'injection-block' | 'request' | 'response';
type LoggingSink = 'stdout' | 'file' | 'syslog' | 'http-collector';
```

**Enforcement:** the gateway *is* the log point, so this compiles natively. Per schema
description it is full on **kong / coraza / envoy / bunkerweb**
(coraza `SecAuditLog`/`SecAuditLogParts` + per-event tagged log actions, bunkerweb modsec
audit config, kong `http-log`/`file-log`/`tcp-log`, envoy `access_log` JSON format).
Other targets: see capability matrix.

## 4. `response.forbidArrayRoot` — JSON-hijacking defense

**Field path:** `response.forbidArrayRoot` (on `ResponsePolicy`).
**Introduced:** v0.7. **Type:** `boolean` (default `false`).
**OWASP:** `API3:2023` (listed in `owasp-mapping.json` under API3's `mitigatedBy`).

Rejects a bare top-level array response body (the classic JSON-hijacking shape).

## 5. `request.idempotencyKey` — replay / double-submit defense

**Field path:** `request.idempotencyKey` (on `RequestPolicy`).
**Introduced:** v0.7. **OWASP:** `API6:2023` (in API6's `mitigatedBy`).

**Type — `IdempotencyKey`:**
```ts
interface IdempotencyKey {
  header: string;   // client idempotency-key header, e.g. 'Idempotency-Key'
  ttl: Duration;    // a repeated key within this TTL is treated as a replay
}
```

**Enforcement:** **partial** — stops replay / double-submit, **not** in-handler races
(per schema/type description).

## 6. `request.serializeBy` + `request.concurrencyLimit` — live-concurrency serialization

**Field paths:** `request.serializeBy`, `request.concurrencyLimit` (on `RequestPolicy`).
**Introduced:** v0.8 (deferred-residuals). **OWASP:** `API6:2023`
(`request.serializeBy` is in API6's `mitigatedBy`).

**Types:**
```ts
interface SerializeBy {
  key: string;                        // RuleRef-style, e.g. 'request.body.account_id' | 'jwt.sub'
  scope?: 'global' | 'per-identifier';
}
// request.concurrencyLimit: number   // max in-flight for the serializeBy key (1 == strict serialize)
```

**Enforcement:** **PARTIAL AT BEST** — edge serialization of same-key requests only; the
gateway **cannot** make the app handler transaction atomic. Per the type doc: coraza /
bunkerweb partial (crude `SecCollection` cap, not a real mutex), envoy override-only
(operator-supplied limiter), kong / aws-apigw / cloudflare / firewall / openappsec
unsupported. **Never full.** Exact cells: see capability matrix.

## 7. `request.dataAtRest` — at-rest storage posture (ADVISORY-ONLY)

**Field path:** `request.dataAtRest` (on `RequestPolicy`).
**Introduced:** v0.8 (deferred-residuals). **SSEC category:** `SSEC-STORAGE`.

**Type — `DataAtRest`:**
```ts
interface DataAtRest {
  fields: string[];
  protection: 'encrypted' | 'hashed' | 'tokenized';
}
```

**Enforcement:** **ADVISORY-DECLARATION ONLY — NOT gateway-enforced.** It compiles to
**nothing** enforcing (the gateway never sees the DB write). Capability is hard-pinned
override-only/unsupported on every target (never full, never partial). It documents the
declared posture and drives an **out-of-band `SSEC-STORAGE` scan finding** — a finding,
not a control.

## 8. `graphql.operations[]` — per-operation GraphQL policy

**Field path:** `graphql.operations[]` (on `GraphqlPolicy`).
**Introduced:** v0.8 (deferred-residuals). **OWASP:** per-op `authz` is in `API1:2023`
and `API5:2023` `mitigatedBy` as `graphql.operations.authz`.

**Type — `GraphqlOperation`:**
```ts
interface GraphqlOperation {
  name: string;                                  // required
  operationType?: 'query' | 'mutation' | 'subscription';
  authz?: Authorization;                         // reuses the Authorization grammar
  maxDepth?: number;
  maxComplexity?: number;
  maxAliases?: number;
}
```

Binds a per-operation policy to a named field/operation the single `POST /graphql` route
funnels. `authz` reuses the `Authorization` grammar (rule-based
`jwt.sub == resource.owner` → `API1:2023` BOLA; `rbac` → `API5:2023` BFLA). Per-operation
cost limits **override** the block-level coarse limits.

**capKey separation:** per-op authz is `graphql.operations.authz`, **distinct** from the
coarse block-level `graphql.staticLimits` (`API4:2023`). Block-level limits stay coarse;
`operations[]` is additive.

**Enforcement:** **OVERRIDE-ONLY on every target** — a gateway cannot evaluate
per-resolver authz/cost without an operator-supplied GraphQL-aware processor. x-security
emits scaffolding only; enforcement depends on the operator supplying the processor.

## 9. `authentication.passwordPolicy` + `authentication.accountLockout`

**Field paths:** `authentication.passwordPolicy`, `authentication.accountLockout`
(on `Authentication`).
**Declared:** v0.5 (S-12); **enforcement capability wired:** v0.7.
**OWASP:** both in `API2:2023` `mitigatedBy`.

**Types:**
```ts
interface PasswordPolicy {
  minLength?: number;
  requireUppercase?: boolean;
  requireDigit?: boolean;
  requireSymbol?: boolean;
  blocklist?: string[];
}
interface AccountLockout {
  attempts: number;
  window: Duration;
  identifier: string;   // e.g. 'header:X-Username' | 'request.body.email'
}
```

`accountLockout` = per-identifier brute-force lockout; `passwordPolicy` = strength policy
for body-carried passwords. Per-target enforcement: see capability matrix.
