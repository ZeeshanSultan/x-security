# x-security schema v0.3 — additions

Schema id: `https://usewaf.com/schemas/x-security/v0.3.json`
Status: design doc for downstream compilers (`compiler-cloudflare`, `compiler-aws-apigw`, `scope-doc-updater`).

v0.3 closes the major web-vuln-class gaps that v0.2 did not cover: CSRF, response-side hardening headers, JWT algorithm confusion, BOLA via resource-aware authorization rules, HMAC webhook verification, GraphQL/WebSocket policies, and CAPTCHA-style bot protection.

All additions are additive. Existing v0.2 policies continue to validate **with one exception**: `authentication.type: bearer-jwt` now requires `allowedAlgorithms`. The v0.2 test fixture using bearer-jwt was updated in-place because the JWT-algorithm-confusion gap (`alg: none`, HS-vs-RS confusion) is exactly the class of bug v0.3 is meant to eliminate; emitting any bearer-jwt policy without an algorithm allowlist would defeat the addition. Compilers should treat this as a hard schema-level requirement, not a recommendation.

For each addition: field path, type, semantics, example, OWASP mapping, and gateway primitive hints.

---

## Authorization / authentication (5)

### 1. PW-1 — `RuleRef` type and `Authorization.rules[].value` widening

**Field paths:** `authorization.rules[].value` accepts `RuleRef = { ref: "(jwt|request|resource).<path>" }` in addition to scalars and arrays. New `$defs/RuleRef`.

**Type:**
```ts
type RuleRef = { ref: string }; // pattern ^(jwt|request|resource)\.[A-Za-z0-9_.]+$
type AuthorizationRuleValue = string | number | boolean | Array<...> | RuleRef;
```

**Semantics:** Lets a rule compare two run-time values instead of comparing a request value against a constant. The canonical use is BOLA defense: `resource.ownerId == jwt.sub`. The compiler resolves `jwt.*` from the verified token, `request.*` from headers/params/body, and `resource.*` from the result of `authorization.resourceLookup` (see #3). Without #3, only `jwt.*` and `request.*` are reachable.

**Example:**
```yaml
authorization:
  type: rule-based
  rules:
    - field: resource.ownerId
      operator: equals
      value: { ref: jwt.sub }
```

**OWASP:** API1:2023 (BOLA), API3:2023 (BOPLA).

**Gateway primitives:**
- Cloudflare: Workers script that reads the JWT after `Access` validation and rejects mismatch; `wirefilter` cannot dereference JSON claims directly, so this requires a custom Worker.
- AWS API Gateway: Lambda authorizer that emits the policy + a context map; the rule comparison happens in a downstream Lambda or VTL mapping template. REST APIs can express it via request validator + custom authorizer context; HTTP APIs need a Lambda.
- Kong / Coraza: existing v0.2 path, now with ref resolution in the plugin.

### 2. `Authentication.allowedAlgorithms` (required for `bearer-jwt`)

**Field path:** `authentication.allowedAlgorithms`

**Type:** `JwtAlgorithm[]` where `JwtAlgorithm = 'RS256'|'RS384'|'RS512'|'ES256'|'ES384'|'ES512'|'EdDSA'|'PS256'|'PS384'|'PS512'`. HS\* and `none` are intentionally absent.

**Semantics:** Hard allowlist of JWT signing algorithms. The enum is asymmetric-only by construction, eliminating both `alg: none` and the classic HS256-with-RS256-public-key confusion. Required whenever `type === 'bearer-jwt'`.

**Example:**
```yaml
authentication:
  type: bearer-jwt
  jwksUri: https://auth.example.com/.well-known/jwks.json
  allowedAlgorithms: [RS256, ES256]
```

**OWASP:** API2:2023 (Broken Authentication), API8:2023 (Security Misconfiguration).

**Gateway primitives:**
- Cloudflare: `Access` JWT validator algorithm whitelist; or in a Worker via `jose` `jwtVerify(token, key, { algorithms })`.
- AWS API Gateway: JWT authorizer's `IdentitySource` plus an explicit algorithm check in the authorizer (HTTP API JWT authorizers do not natively gate on `alg`).
- Kong: `jwt`/`openid-connect` plugins' `algorithms` field.

### 3. `Authorization.resourceLookup`

**Field path:** `authorization.resourceLookup`

**Type:**
```ts
{ endpoint: string; identifierFrom: string; expose: string[] }
```

**Semantics:** Before evaluating `rules`, the gateway calls `endpoint` (a path template) with the identifier extracted from `identifierFrom` (e.g. `request.params.id`), parses the response, and exposes the listed fields under the `resource.*` namespace. This is what makes RuleRef-based BOLA checks practical: rules can ask "does the JWT subject match the resource owner?" without bespoke handler code.

**Example:**
```yaml
authorization:
  type: rule-based
  resourceLookup:
    endpoint: /users/{id}
    identifierFrom: request.params.id
    expose: [ownerId, tenantId]
  rules:
    - field: resource.ownerId
      operator: equals
      value: { ref: jwt.sub }
```

**OWASP:** API1:2023 (BOLA), API5:2023 (BFLA).

**Gateway primitives:**
- Cloudflare: Worker pre-fetch (subrequest) to the lookup endpoint, attach to `event.data`.
- AWS API Gateway: Lambda authorizer issues a sigv4 sub-call to the lookup endpoint (or a DynamoDB read) and stores results in `context`.
- Kong: pre-function plugin or a small Lua plugin doing the sub-request.

### 4. `csrf` (top-level)

**Field path:** `csrf`

**Type:**
```ts
{
  method: 'origin-check' | 'double-submit' | 'custom-header';
  allowedOrigins?: string[];   // required if method=origin-check
  tokenHeader?: string;        // required if method=double-submit | custom-header
  tokenCookie?: string;        // required if method=double-submit
}
```

**Semantics:** Declarative CSRF defense. `origin-check` validates `Origin`/`Referer` against an allowlist (cheap, works for SPAs with strict CORS). `double-submit` verifies a token in both a header and a cookie. `custom-header` requires presence of an arbitrary header (relies on browser CORS to block cross-origin sets).

**Example:**
```yaml
csrf:
  method: double-submit
  tokenHeader: X-CSRF-Token
  tokenCookie: csrf_token
```

**OWASP:** API8:2023 (Security Misconfiguration). Web/CWE-352.

**Gateway primitives:**
- Cloudflare: ruleset rule matching `http.request.headers["origin"] not in {allowedOrigins}` → block; or Worker reading the cookie+header pair.
- AWS API Gateway: request validator + WAFv2 custom rule (string-match on `Origin`); double-submit needs a Lambda authorizer.
- Kong: `request-transformer` + custom plugin; BunkerWeb has a CSRF module.

### 5. `response.cookies.defaults`

**Field path:** `response.cookies.defaults`

**Type:**
```ts
{ httpOnly?: boolean; secure?: boolean; sameSite?: 'Strict'|'Lax'|'None';
  path?: string; domain?: string; maxAge?: number }
```

**Semantics:** Applied to every `Set-Cookie` header emitted on this endpoint's response. Acts as a "fill-missing-attributes" filter — if the origin already set `HttpOnly`, the gateway leaves it; if it didn't, the gateway appends. The gateway never weakens existing attributes.

**Example:**
```yaml
response:
  cookies:
    defaults: { httpOnly: true, secure: true, sameSite: Strict }
```

**OWASP:** API8:2023.

**Gateway primitives:**
- Cloudflare: Transform Rule on response headers (Set-Cookie rewrite) or Worker.
- AWS API Gateway: gateway response mapping template (REST) or response parameter mapping (HTTP).
- Kong: `response-transformer-advanced`.

## Request hardening (7)

### 6. PW-2 — `request.denyUnknownFields`

**Field path:** `request.denyUnknownFields` (boolean, default `false`)

**Semantics:** When true, the gateway rejects any request body field outside the keys of `request.schema` with 400. Auto-generated policies for `POST/PUT/PATCH` should set this to `true`. Default-false preserves v0.2 compatibility.

**Example:**
```yaml
request:
  denyUnknownFields: true
  schema:
    id: { type: uuid }
    name: { type: string, maxLength: 200 }
```

**OWASP:** API3:2023 (BOPLA / mass assignment), API6:2023 (unrestricted resource consumption via oversized payloads).

**Gateway primitives:**
- Cloudflare: Worker with JSON schema validator (ajv-compiled).
- AWS API Gateway: model + request validator (`Validate body, query string parameters, and headers`) with `additionalProperties: false`.
- Kong: `request-validator` plugin.

### 7. `request.signature`

**Field path:** `request.signature`

**Type:**
```ts
{
  algorithm: 'hmac-sha256' | 'hmac-sha1' | 'ed25519';
  headerName: string;
  secretRef: VarRef;
  body: 'raw' | 'canonical';
  timestampHeader?: string;
  timestampToleranceSeconds?: number; // 1..3600
}
```

**Semantics:** Webhook signature verification at the edge before the request reaches origin. `body: canonical` JSON-canonicalizes before HMAC; `raw` uses the byte stream. A `timestampHeader` + tolerance pair adds replay protection.

**Example (Stripe-style):**
```yaml
request:
  signature:
    algorithm: hmac-sha256
    headerName: Stripe-Signature
    secretRef: $vault.webhooks/stripe
    body: raw
    timestampHeader: Stripe-Timestamp
    timestampToleranceSeconds: 300
```

**OWASP:** API2:2023, API8:2023, API10:2023 (Unsafe Consumption of APIs — inbound side).

**Gateway primitives:**
- Cloudflare: Worker computing HMAC with `crypto.subtle`.
- AWS API Gateway: Lambda authorizer (HMAC) — note: API Gateway has no native HMAC primitive.
- Kong: `hmac-auth` plugin (close fit) or a custom Lua plugin for Stripe-style schemes.

### 8. `request.allowedHosts`

**Field path:** `request.allowedHosts`

**Type:** `string[]`

**Semantics:** Host-header allowlist. Defends against host-header injection and cache-poisoning attacks routed via `Host:` rewrites.

**Example:**
```yaml
request:
  allowedHosts: [api.example.com, api-eu.example.com]
```

**OWASP:** API8:2023.

**Gateway primitives:**
- Cloudflare: ruleset `http.host in {...}` → otherwise block.
- AWS API Gateway: custom domain mapping + WAFv2 host match rule.
- Kong: route `hosts` field is already the natural fit.

### 9. `request.duplicateParamPolicy`

**Field path:** `request.duplicateParamPolicy` (enum `first | last | reject`, default `reject`)

**Semantics:** HTTP Parameter Pollution (HPP) defense. The gateway picks one canonical value (or rejects), so downstream services see a single deterministic value.

**Example:**
```yaml
request: { duplicateParamPolicy: reject }
```

**OWASP:** API8:2023, web CWE-235.

**Gateway primitives:**
- Cloudflare: Worker normalization of `URL.searchParams`.
- AWS API Gateway: not natively expressible — emit as Lambda authorizer/integration request mapping note.
- Kong / Coraza: ModSecurity rule (Coraza has CRS HPP rules built-in).

### 10. `request.headerInjectionGuard`

**Field path:** `request.headerInjectionGuard` (boolean, default `false`)

**Semantics:** Reject any request whose header values contain `\r`, `\n`, or `\0`. Defends against HTTP request smuggling and header-injection-based response splitting.

**Example:**
```yaml
request: { headerInjectionGuard: true }
```

**OWASP:** API8:2023, web CWE-93/CWE-113.

**Gateway primitives:**
- Cloudflare: managed ruleset already covers most of this; pin a custom rule to make it explicit per route.
- AWS API Gateway: WAFv2 regex-match rule on common header fields.
- Kong / Coraza: CRS rule 921110/921120.

### 11. `request.pathCanonicalization`

**Field path:** `request.pathCanonicalization` (boolean, default `false`)

**Semantics:** Decode percent-encoding, resolve `.` / `..`, and collapse `//` in the URL path before any pattern check (CORS origin checks, route match, WAF rules). Prevents bypass via `/admin/%2e%2e/users`, `/admin/..;/users`, double-slash `//admin`, etc.

**Example:**
```yaml
request: { pathCanonicalization: true }
```

**OWASP:** API1:2023 (auth bypass via path tricks), API8:2023.

**Gateway primitives:**
- Cloudflare: built-in; expressed as a transform-rule setting (`http.request.uri.path` normalization is the default but should be asserted).
- AWS API Gateway: requires CloudFront in front for canonicalization; flag as a compiler warning when CloudFront is absent.
- Kong: `path_handling` v1 plus a small pre-function.

### 12. `ParamSchema` binary upload fields

**Field paths:** `request.schema.<param>.magicByteCheck`, `.extensionAllowlist`, `.denyDoubleExtension`

**Types:** `boolean`, `string[]` (each item `^\.[A-Za-z0-9]+$`), `boolean`.

**Semantics:** Anti-MIME-spoofing for file uploads. `magicByteCheck` verifies the declared `allowedMimeTypes` against the file's actual signature bytes. `extensionAllowlist` constrains the filename suffix. `denyDoubleExtension` blocks `invoice.pdf.exe`-style filenames.

**Example:**
```yaml
request:
  schema:
    avatar:
      type: binary
      allowedMimeTypes: [image/png, image/jpeg]
      maxSize: 2MB
      magicByteCheck: true
      extensionAllowlist: [.png, .jpg, .jpeg]
      denyDoubleExtension: true
```

**OWASP:** API8:2023, API10:2023.

**Gateway primitives:**
- Cloudflare: Worker (R2 or buffered body) sniffs first 8–16 bytes; reject on mismatch.
- AWS API Gateway: binary support + Lambda authorizer/pre-processor; payload size cap (10MB) limits viability — flag for compiler if `maxSize > 10MB`.
- Kong: `file-log` is irrelevant — needs a custom plugin or upstream service.

## Response / cache (2)

### 13. `response.headers` (highest-leverage addition)

**Field path:** `response.headers`

**Type:** see schema. All sub-fields optional. Absence means "do not emit."

**Semantics:** Single declarative knob for CSP, HSTS, X-Frame-Options, X-Content-Type-Options, Referrer-Policy, Permissions-Policy, COOP/COEP/CORP, and Cache-Control override. The gateway emits exactly the requested headers; it does not synthesize defaults.

**Example:**
```yaml
response:
  headers:
    csp: "default-src 'self'; script-src 'self'; object-src 'none'"
    hsts: { maxAge: 31536000, includeSubDomains: true, preload: true }
    frameOptions: DENY
    contentTypeOptions: nosniff
    referrerPolicy: strict-origin-when-cross-origin
    permissionsPolicy: "camera=(), microphone=()"
    coop: same-origin
    coep: require-corp
    corp: same-origin
    cacheControl: no-store
```

**OWASP:** API8:2023 (catch-all). Web/OWASP Top 10 A05.

**Gateway primitives:**
- Cloudflare: Transform Rules → "Modify Response Header" for each field.
- AWS API Gateway: Gateway Response with header overrides (REST); response parameter mapping (HTTP).
- Kong: `response-transformer` per-header.

### 14. `Cacheable.unkeyedHeadersStrip`

**Field path:** `cacheable.unkeyedHeadersStrip` (only on the object form)

**Type:** `string[]`

**Semantics:** Headers to strip from the request **before computing the cache key**. Critical defense against cache poisoning via headers that influence origin behavior but are unsafe to vary on (e.g. `Cookie`, `Authorization`, `X-Forwarded-Host`).

**Example:**
```yaml
cacheable:
  enabled: true
  ttl: 60
  unkeyedHeadersStrip: [Cookie, Authorization, X-Forwarded-Host]
```

**OWASP:** API8:2023, web CWE-444 / cache-poisoning.

**Gateway primitives:**
- Cloudflare: Cache Rules → "Remove header from cache key."
- AWS API Gateway: native API GW cache is keyed on path only; this maps to CloudFront cache policy `headersConfig` if CloudFront is in front. Otherwise emit a compiler warning.
- Kong: `proxy-cache` `vary_headers` is the inverse — the compiler should emit a non-vary-list and rely on Coraza for true stripping.

## Protocol-specific (2)

### 15. `graphql`

**Field path:** `graphql` (top-level)

**Type:**
```ts
{
  maxDepth?: number;       // 1..50
  maxComplexity?: number;
  maxAliases?: number;
  batchLimit?: number;
  disableIntrospection?: boolean;
  allowedOperations?: ('query'|'mutation'|'subscription')[];
}
```

At least one limit must be set (`minProperties: 1`).

**Semantics:** GraphQL-specific abuse defenses. Depth and complexity limits stop quadratic explosion; alias limits stop alias-based DoS; batch limit stops batched-query brute-forcing; introspection toggle hides the schema in prod; operation allowlist disables mutations on read-only edges.

**Example:**
```yaml
graphql:
  maxDepth: 10
  maxComplexity: 1000
  maxAliases: 15
  batchLimit: 10
  disableIntrospection: true
  allowedOperations: [query, mutation]
```

**OWASP:** API4:2023 (Unrestricted Resource Consumption), API6:2023, API8:2023.

**Gateway primitives:**
- Cloudflare: Worker with graphql-armor or hand-rolled AST walk; no native primitive.
- AWS API Gateway: AppSync's `EvaluationRules` partially covers this; for raw API GW, Lambda only.
- Kong: `graphql-rate-limiting-advanced` + `graphql-proxy-cache-advanced` (enterprise) or a custom plugin.

### 16. `websocket`

**Field path:** `websocket` (top-level)

**Type:**
```ts
{
  allowedOrigins: string[];                       // required
  maxMessageSize?: ByteSize;
  messageRateLimit?: { messages: number; window: Duration };
  maxConnectionsPerIdentifier?: number;
  idleTimeout?: Duration;
}
```

**Semantics:** Per-endpoint WebSocket controls. Origin allowlist on the upgrade handshake; per-message size cap; per-connection message rate; per-identifier connection cap; idle timeout for dead-socket cleanup.

**Example:**
```yaml
websocket:
  allowedOrigins: [https://app.example.com]
  maxMessageSize: 64KB
  messageRateLimit: { messages: 100, window: 1s }
  maxConnectionsPerIdentifier: 5
  idleTimeout: 5m
```

**OWASP:** API4:2023, API8:2023.

**Gateway primitives:**
- Cloudflare: Durable Object + Worker; `allowedOrigins` is the only handshake-level concern.
- AWS API Gateway: WebSocket APIs have native idle timeout (10m max); message size limit is 128KB hard cap; rate-limit is per-route via usage plan.
- Kong: needs custom plugin; not native.

### 17. `botProtection`

**Field path:** `botProtection` (top-level)

**Type:**
```ts
{
  provider: 'turnstile' | 'recaptcha' | 'hcaptcha';
  secretRef: VarRef;
  threshold?: number; // 0..1, default 0.5
  mode: 'enforce' | 'observe';
}
```

**Semantics:** CAPTCHA-style bot challenge integration. `observe` mode logs scores without blocking — for tuning the threshold before flipping to `enforce`.

**Example:**
```yaml
botProtection:
  provider: turnstile
  secretRef: $vault.bots/turnstile-secret
  threshold: 0.7
  mode: enforce
```

**OWASP:** API4:2023 (resource abuse), API2:2023 (credential stuffing).

**Gateway primitives:**
- Cloudflare: Turnstile is native; rendered on the client, validated on the server via siteverify. The compiler emits the secret binding and a verification Worker step.
- AWS API Gateway: Lambda authorizer that calls Google/hCaptcha siteverify.
- Kong: `bot-detection` plugin is heuristic-only; needs a custom plugin for CAPTCHA verification.

## Profile taxonomy expansion (post-v0.3)

The `profile` enum was extended with six additional presets to cover endpoint shapes that the original six did not fit: `server-rendered-page`, `graphql-resolver`, `grpc-method`, `internal-rpc`, `static-asset`, and `unknown` (explicit "unclassified" replacing silent `standard-crud` fallback). Profile defaults live in `packages/llm-agent/src/agentic/profiles.ts`.

## Notes for downstream compilers

1. **Backward-compat surface:** every v0.2 policy still validates **except** bearer-jwt policies missing `allowedAlgorithms`. Compilers SHOULD treat that as a hard error at policy-load time and refuse to emit any gateway config for the endpoint.
2. **`TargetOverrides` keys:** v0.3 adds `cloudflare` and `aws-apigw` to the allowed propertyNames so compilers can attach vendor-specific knobs without a schema change.
3. **Capability gaps:** API Gateway lacks native primitives for (a) HMAC signing, (b) HPP, (c) request body field-deny, (d) path canonicalization, (e) GraphQL/WebSocket fine-grained limits, (f) bot-protection verification. The compiler should emit Lambda authorizers / WAFv2 rules / CloudFront policies as needed and surface compiler warnings when a feature degrades to "best-effort."
4. **`$id`** is now `v0.3.json`; consumers pinning to `v0.2.json` should continue to validate against the v0.2 file (kept as historical record in version control).
